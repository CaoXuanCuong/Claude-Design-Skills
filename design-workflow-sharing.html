01 — Vấn đề

Khoảng cách design → code đang ăn thời gian của ta

Quy trình cũ: mockup tĩnh → doc bàn giao → dev tự đọc, tự đoán → code lệch → review qua lại.
handoff-cu.log
→ designer vẽ mockup tĩnh trên Figma
→ viết doc mô tả spacing, state, behavior... (hoặc không)
✗ dev đọc, đoán component nào, state nào, event nào
→ code ra lệch pixel / thiếu state / sai UI components
⟳ review qua lại 3–4 vòng — chi phí nằm ở khâu "dịch" thủ công

Điểm nghẽn không phải "vẽ đẹp" — mà là dịch thiết kế thành code đúng, lặp đi lặp lại bằng tay.
→ Gần đây có một công cụ sinh ra để gỡ đúng khâu này.

05 — Định vị

Ở thời điểm hiện tại, Claude Design chưa thay được Figma. Mỗi bên mạnh một khúc — dùng đúng chỗ mới tối ưu.

  • Pain: handoff → dev vẫn dịch tay sang code, dễ lệch
+
CLAUDE DESIGN · nơi tạo nhanh & nối code
  • Mạnh: design system tái sử dụng — vẽ bằng đúng component (Polaris)
  • Mạnh: sinh UI thật, chạy được; output là bundle máy đọc được → code
  • Mạnh: dựng nhanh nhiều state / phương án để chốt sớm
  • Hạn chế: craft thị giác & pixel-control tinh xảo chưa bằng Figma
  • Hạn chế: collaboration, plugin, quản asset/brand — Figma vẫn hơn

Cách dùng: Figma để chốt hệ thống thị giác & polish · Claude Design để tạo nhanh, thử state & nối mạch sang code.

07 — Quy trình mới

Một dây chuyền từ ý tưởng tới code đã verify

Toàn bộ đóng gói trong /bloy:build-feature --ticket "BLOY-xx" — mỗi step chạy ở context riêng, có state & gate.
B1 · prototype

Dựng & iterate UI trên Claude Design, chốt các state cần có.

B2 · gather-spec

Gộp requirement Jira + bundle design thành một spec.md duy nhất.

B3 · discovery

Map mỗi design-state → route / component / Polaris cần ADD·UPDATE·DELETE.

B4 · build + verify

Plan → code → Playwright screenshot & visual-diff về đúng bản design.

08 — Trái tim của mạch nối

design-discovery: bắc cầu design ↔ code

Đây là bước phân tích (chưa code) — biến bundle design thành bản đồ triển khai chính xác cho CMS admin (Polaris SPA).
Map — không đoán

Design-state → route/component

Mỗi state trong design được gắn vào đúng route & component của app (xác định app-router qua codegraph), và mỗi element → đúng Polaris component.

Chỉ rõ việc phải làm

ADD · UPDATE · DELETE

Liệt kê element nào cần thêm/sửa/xoá, handler/event nào cần wire — thay vì để dev tự suy diễn từ hình.

Đường tới state

Playwright action chain

Chuỗi action để tới đúng state đó và để verify, kèm screenshot/crop tham chiếu.

Đo được

Visual diff + tolerance

Gắn reference screenshot & ngưỡng sai khác → khớp/không khớp là fact, không phải cảm tính.

Output là spec đủ chi tiết để một agent khác code & tự verify frontend — khâu "dịch tay" biến mất.

09 — Cách tiếp cận

Team mình leo 3 bậc, không nhảy cóc

1

Prototype thay cho mô tả (designer + PM)

Dùng Claude Design để dựng giao diện thật thay vì mô tả bằng chữ. Chốt đủ các state (empty · loading · error · success) ngay trên bản prototype — đây là "spec sống".

2

Bundle vào pipeline (1 lệnh)

Export bundle → /bloy:build-feature --ticket. Để gather-spec + discovery lo phần map sang Polaris/route. Dev review spec & plan, không dịch tay từ hình.

3

Verify bằng mắt máy (dev + QA)

Code xong, Playwright chụp state thật trong Shopify admin và so với bản design. Lệch là gate bắt được — không cần soi pixel bằng tay.

Đừng bắt đầu bằng bậc 3. Giá trị lớn nhất là bậc 1–2 làm cho chắc — prototype rõ, bundle sạch.

10 — Ai làm gì

Mỗi vai giữ đúng phần giá trị của mình

Công cụ không thay vai — nó gỡ phần lặp tay để mỗi người tập trung vào phần khó.
Con người vạch ý đồ & ranh giới
Claude Design + pipeline lo phần "dịch"
Designer · ý đồ & hệ thống thị giác PM · state & acceptance Dev · kiến trúc & review QA · verify visual
⛔ ranh giới cứng: không tự đổi API/data-model · không skip green-gate · Polaris theo đúng API version của lớp đang đụng

Designer vẫn quyết thẩm mỹ, dev vẫn quyết kiến trúc — máy chỉ lấy đi khâu phiên dịch thủ công ở giữa.

11 — Khuôn khổ

Tự chủ trong rào cứng

Tối ưu quy trình không có nghĩa bỏ kiểm soát — vẫn giữ các gate của ta.
  • Prototype trên Claude Design để chốt state & flow — không nhảy thẳng vào code.
  • Luôn qua spec.md + explore-spec — bundle không tự thành code.
  • Playwright verify mọi state — visual diff là fact, không phải "nhìn thấy ổn".
  • Đúng Polaris API version theo lớp — Web layer 2026-01, không dùng "latest".
Rào cứng tuyệt đối

Không tự đổi API / data-model · branch riêng · qua green-gate (build + test xanh) · không push khi chưa duyệt.

Security: không dangerouslySetInnerHTML vô cớ · validate input · query scope theo shopId — bundle design không miễn trừ các luật này.

12 — Được gì

Tối ưu nằm ở khâu bị lặp nhiều nhất

Không phải "vẽ nhanh hơn" — mà là 3 thứ ta lấy lại được từ khâu dịch tay giữa design và code.
01 · thời gian

Bớt vòng dịch tay

Máy map design → Polaris / route, dev chuyển từ "phiên dịch từ hình" sang review spec & plan. Bớt hẳn các vòng qua lại.

02 · chất lượng

Bắt lệch sớm

Playwright visual-diff so bản dựng với design ngay khi code xong. Lệch là fact có ngưỡng — lộ ra sớm, không đợi review muộn.

03 · tri thức

Ghi trong artifact

State & flow nằm trong bundle + spec, không kẹt trong đầu một người. Người mới, hay lần sửa sau, đọc lại được.

Tối ưu thật nằm ở chỗ bỏ được vòng dịch tay và bắt lệch sớm — không phải ở tốc độ vẽ.

14 — Mindset

4 tư duy khi dùng Claude Design

Công cụ chỉ mạnh khi đi kèm cách nghĩ đúng — đây là phần quyết định team dùng nó hiệu quả hay không.
01 · vai trò

Đồng đội, không thay ca

Claude Design lo phần dựng & dịch — gu và quyết định thẩm mỹ vẫn là của con người. Bổ trợ Figma, không thay designer.

02 · thứ tự

Design system trước, page sau

Đầu tư vào chuẩn dùng chung (Polaris + skill) mới ăn tiền. Đừng vẽ ad-hoc từng page rồi mỗi nơi một kiểu.

03 · đơn vị tư duy

Nghĩ bằng component & state

Vì output để nối code. Liệt kê đủ state (empty · loading · error · success) quan trọng hơn là canh lề cho đẹp.

04 · vị thế output

Prototype để nghĩ, không để "xong"

Dùng để thử nhanh & chốt sớm, rẻ. Bundle là nguyên liệu — luôn qua spec + verify, không copy mù ra production.

Đổi cách nghĩ: từ "vẽ giao diện" → "định nghĩa hệ thống, rồi để máy dựng & nối code".
